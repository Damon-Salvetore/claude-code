# CLAUDE.md — 全局开发原则

## 核心原则

**不出错比做完更重要。慢一点没关系，重点是写一定正确的代码、做一定正确的操作。不要急着完成任务。**

## 代码开发

1. **写一定正确的代码。** 不要写"大概能跑"的代码。写之前想清楚每一行的行为，特别是边界情况。一行错误的 `export MAX_UPDATES=5`（应该是 `${MAX_UPDATES:-5}`）就能让 12 个实验全部白跑。

2. **改一处，验一处。** 每改一个东西，立刻验证它是否生效、是否符合预期。不要一次改多处然后祈祷都没问题。改了参数就检查输出，改了路径就确认文件存在。

3. **测试通过的东西原样使用。** 测试用的是哪个脚本、哪些参数、什么环境，正式跑就用一模一样的，只改必须改的部分。不要换一个"更好的"但没测过的方案。

4. **引入的每个新东西都是风险。** 新工具、新脚本、新参数、新的自动化——每一个都需要验证。如果要加，先验证它不会跟现有流程冲突。

5. **小文件优于大文件。** 200-400 行为宜，800 行封顶。功能拆分清晰，每个文件职责单一。

6. **修实现，不修测试。** 测试失败时，先假设实现有 bug。除非测试本身逻辑错误，否则不要为了通过测试去改测试。

7. **增量修错。** 一次只修一个错误，修完验证，再修下一个。不要一口气改一堆然后看结果。

8. **用具体例子说明规则。** 在 CLAUDE.md 里写规则时，不要只说"处理好错误"——给出正确和错误的代码对比。抽象的规则容易被忽略，具体的例子不会。

## 问题排查

9. **出了问题先定位根因，不要急着修。** 反复重试同一个操作不会让 bug 消失。停下来，追溯问题的源头（比如用 `ps -p <pid> -o ppid` 追父进程），找到根因再修。

10. **环境干净和逻辑正确同等重要。** 代码逻辑对了不够，还要确认运行环境没有残留进程、旧缓存、旧日志。很多"莫名其妙的 bug"都是脏环境导致的。

11. **先跑通，再跑对，再跑干净。** Get it working → Get it right → Get it clean。不要在还没跑通的时候就去优化代码结构。

## 流程管理

12. **做一件事就做一件事。** 不要一边跑实验一边改脚本一边启动监控。多个操作同时进行会互相干扰，出了问题也更难定位。

13. **后台任务要知道怎么彻底停止。** 启动 nohup 后台脚本时记录根进程 PID。停止时必须先杀根进程，再杀子进程。只杀子进程是无用功，根进程会不断拉起新的。

14. **长时间任务之前先做端到端验证。** 不要只检查语法和逻辑就投入长时间运行。用最小规模（如 5 步训练）跑一遍完整流程，确认从启动到完成到衔接都没问题。

15. **GPU 训练必须在持久会话中运行。** 用 nohup/tmux/screen 启动长时间训练，永远不要在前台裸跑。记录根进程 PID，方便事后管理。

## 上下文与会话管理

16. **上下文是最宝贵的资源。** 不要在对话中加载不需要的文件、启动不需要的工具。MCP 工具过多会把 200k context 压缩到 70k。保持工具精简。

17. **手动 compact，不要自动。** 自动 compact 会在任务中间触发，丢失关键上下文。在逻辑边界处（一个任务完成后、探索结束准备实现前）手动 `/compact`。

18. **上下文最后 20% 只做低风险操作。** 上下文快用完时，只做单文件编辑、写文档、简单 bug 修复。不要在这个阶段开始复杂的多文件改动。

19. **compact 前先总结状态。** 在 `/compact` 之前，把当前进展、待办事项、关键发现写到 CLAUDE.md 或会话文件里，避免 compact 后丢失关键信息。

## 子 agent 使用原则

20. **子 agent 也是改动，也需要验证。** 不要未经测试就在正式流程旁边启动子 agent。先在测试环境验证子 agent 不会干扰主流程。

21. **给子 agent 明确的目标上下文。** 子 agent 只知道你传给它的 prompt，不知道你的目的。把"为什么要做"和"做什么"都写清楚。

22. **按任务复杂度选 agent 模型。** 探索/搜索用 Haiku（快且便宜），简单编辑用 Haiku，多文件实现用 Sonnet，复杂架构/调试用 Opus。默认 Sonnet，失败了再升 Opus。

23. **子 agent 按职责限定工具。** 规划 agent 只需要 Read/Grep/Glob（不给写权限），代码审查 agent 只需要只读工具。工具越少，agent 越专注。

## 会话与记忆管理

24. **会话文件跨对话传递上下文。** 每次 compact 前或会话结束前，把进展/失败/下一步写到 `.claude/sessions/YYYY-MM-DD-<topic>.tmp`。下次会话 `claude --system-prompt "$(cat file)"` 或提供路径续接。格式：
    - 什么方案试过了（成功/失败+原因）
    - 当前状态（跑到哪了、什么在后台）
    - 下一步要做什么
    - 需要加载哪些文件

25. **按任务切换工作模式。** 研究模式（先读后写、不急着改）→ 开发模式（改完立测、commit 原子性）→ 运维模式（保守操作、每步确认）。NEVER 在运维模式中 "顺便" 改代码。

26. **迭代检索，不要一次猜全。** 子 agent 搜代码时：先广搜 → 评估相关性 → 精炼关键词 → 再搜。最多 3 轮。第一轮经常能发现项目的命名习惯和术语。

## ML 训练 & 模型压缩通用原则

27. **训练启动前必须有验证 checklist。** GPU 干净、无残留进程、缓存已清、参数已确认。不要 "应该没问题" 就跑。一次错误的超参可能浪费数小时 GPU 时间。

28. **先跑极短步数做端到端验证。** 无论 PTQ calibration 还是 QAT 训练，先跑 5-10 步验证整个 pipeline：数据加载 → forward → loss 计算 → backward → optimizer step → logging → 保存/衔接。长时间任务前必须端到端通过。

29. **量化实验必须隔离变量。** PTQ 比较不同量化方案时，只改量化配置，不改模型架构/数据/超参。QAT 比较时同理。所有 baseline 和实验组用相同的 seed、数据顺序、学习率。

30. **训练进程的生命周期不可控时用轮询，不用 wait。** torchrun/deepspeed 的子进程经常不正常退出。用日志标记（如 wandb sync 完成）+ 超时保护代替 `wait $PID`。

31. **GPU 显存释放需要时间。** kill 进程后 GPU 显存不会立即释放。`kill -9` 后 `sleep 5`，再检查 `nvidia-smi`。连续检查两次确认干净。

32. **wandb/tensorboard 是实验的唯一可信记录。** 不要靠记忆或终端输出判断实验结果。所有对比必须从 wandb API 或日志文件提取数值。每批实验用新 project name 隔离。

33. **checkpoint 和量化参数的精度敏感。** PTQ 的 scale/zero_point 存储精度、QAT 的 fake quant 节点放置位置，一点偏差就会导致精度崩塌。修改量化代码后必须验证：forward 输出与修改前 diff 是否在预期范围内。

34. **分布式训练脚本的进程管理。** ALWAYS 记录根进程 PID。停止时先杀根（nohup shell），再杀 launcher（torchrun），最后杀 worker。反过来无用，根进程会不断拉起新的。

35. **模型压缩的核心指标三角：精度-速度-大小。** 任何量化方案都要同时报告：(1) task accuracy/loss, (2) inference throughput, (3) model size/memory。只报其中一个没有意义。

## 项目级 CLAUDE.md 写法建议

36. **结构清晰：项目概述 → 硬性规则 → 文件结构 → 代码模式 → 环境变量 → 常用命令。** 让 Claude 快速了解项目全貌。

37. **硬性规则用 ALWAYS/NEVER/MANDATORY 标注。** 模糊的建议会被忽略，明确的禁令不会。

38. **包含文件结构树。** 一个简短的 ASCII 目录树让 Claude 不用探索就知道代码在哪。

39. **包含成功标准。** 写清楚"什么算完成"：所有测试通过、无安全漏洞、代码可读、性能达标。

40. **包含验证 checklist。** 尤其是 GPU 训练项目，写清楚启动前要检查什么、训练完成的判断标准、异常处理步骤。
